今日修改要求整理（M3）

用途
本文件用于在版本回退后，重新实现今天已经明确提出过的 M3 修改需求。
内容按“用户原始要求 + 实际应落地的功能点”整理，便于直接照单重做。


一、主题区与命名调整

1. 文案调整
- 将“经典万花筒”改为“经典”。
- 相关显示名可统一整理为：
  - 经典
  - 经典主题
  - 填充子主题
  - 连线子主题
  - 柔体主题

2. 主题启用方式调整
- 主题的启用，不再放在右侧参数面板里重复出现。
- 主题的启用，改为在左侧“主题设置”树形结构中完成。
- 左侧主题树的每一行右侧都要有一个小复选框，用来表示该主题是否启用。

3. 主题树结构要求
- 左侧保留“主题设置”节点。
- 主题结构应支持：
  - 经典
  - 填充子主题
  - 连线子主题
  - 柔体主题
- 经典主题本身是一个总开关。
- 填充子主题和连线子主题可以分别启用和关闭。
- 柔体主题单独启用和关闭。

4. 启用逻辑要求
- 经典主题总开关开启时，应同时影响经典下的填充和连线。
- 填充子主题控制 contours_enabled 一类的状态。
- 连线子主题控制 bars_enabled 一类的状态。
- 柔体主题控制 tentacles_enabled 一类的状态。
- 左侧树上的小复选框状态，必须与当前配置实时同步。

5. 右侧页面说明调整
- 右侧对应主题详情页，不再重复放置“启用”复选框。
- 改为用提示文本说明：启用请在左侧树中完成。
- 右侧页面只负责参数调节、子预设管理、说明文字。


二、预设管理调整

1. 预设切换方式修改
- 预设切换不应只靠当前下拉框选择。
- 增加“上一个”“下一个”两个按钮。
- 自动切换不再随机选另一个预设，而应按当前预设顺序顺序切换。

2. 自动切换文案调整
- “自动随机切换”改为“自动切换”。

3. 自动切换逻辑
- 自动切换应按照当前预设列表顺序依次切换。
- 切换到末尾后回到开头循环。
- 如果开启自动切换，则按设定时间间隔继续调度。

4. 预设文件可用性检查
- 现有 presets 目录和 section_presets 目录中的 JSON 预设文件，需要保证加载时不会报错。
- 需要在刷新预设列表时对预设文件进行验证。
- 如果某个预设文件无效：
  - 不应让整个预设列表崩掉。
  - 应跳过该预设。
  - 信息栏提示已跳过多少个无效预设。

5. 预设加载兼容性要求
- 预设加载要统一走一个兼容入口。
- 需要兼容已有预设文件结构。
- 需要兼容后续增加的主题混合数据结构，例如：
  - _preset_meta
  - _theme_mix

6. 预设分类相关
- 预设分类逻辑如果保留，应保证不影响已有预设切换与排序。
- “全部预设 / 原有主题 / 水母主题”这类分类，只能作为筛选增强，不能破坏原来的预设顺序与兼容加载。


三、已出现过的运行错误修复项

1. seg_spin 缺失错误
- 曾出现运行时报错：
  'VisualizerControlUI' object has no attribute 'seg_spin'
- 需要确保控制区中的“段数”控件存在。
- 该控件应正确绑定 circle_segments。


四、窗口右上角新增功能

1. 新增三个功能按钮
- 顶部右上角加入：
  - 置于顶层
  - 置于底层
  - 重启程序

2. 置于顶层/底层逻辑
- 需要真正影响窗口层级，而不是只改一个布尔值显示。
- 建议配置中支持 window_layer，取值至少包括：
  - top
  - normal
  - bottom
- 顶层、普通、底层状态应与 UI 保持同步。

3. 重启程序逻辑
- “重启程序”应尽量重启整个主程序，而不是只重启局部渲染器进程。
- 点击后应重新拉起主程序，再关闭当前进程。


五、颜色显示块同步更新

1. 同步要求
- 所有颜色显示块必须同步更新。
- 包括但不限于：
  - 手动改颜色时
  - 预设切换时
  - 预设切换的中间过渡时

2. 预设切换过渡期间
- 在预设平滑过渡过程中，颜色预览块也必须实时反映当前过渡后的颜色状态。
- 不能只在过渡结束时才刷新颜色块。

3. 实现方向
- 渲染端应能把运行时颜色状态回传给主 UI。
- 主 UI 需要接收并应用这份运行时颜色快照。


六、柔体主题参数加入平滑过渡

1. 当前要求
- 柔体主题的参数在预设切换时，不能突变。
- 需要加入平滑过渡规则。
- 数值变化应为缓动过渡。

2. 过渡范围
- 不仅颜色要过渡。
- 柔体参数本身也要参与过渡，例如：
  - 长度
  - 数量
  - 扰动
  - 摆动
  - shader 相关参数
  - 中心驱动相关参数


七、经典填充与连线参数重构为右键 K/P 绑定样式

1. 总体目标
- 原先经典主题中的填充和连线参数，需要改成和柔体主题类似的右键绑定样式。
- 即：参数标签支持右键菜单绑定 K / P。
- 绑定区样式与柔体主题保持一致。

2. 重构原则
- 原先已经“内建绑定”的效果，不应继续直接写死在单一数值里。
- 要拆成：
  - 基础值
  - 是否绑定 K / P
  - K / P 权重范围

3. 需要支持 K/P 绑定的经典参数
- 填充层至少包括：
  - 厚度
  - 透明度
  - 填充透明度
  - 间隔
  - 衰减
  - 转速
  - rot_pow
- 连线层至少包括：
  - 厚度
  - 固定长度

4. UI 要求
- 参数名标签支持右键菜单。
- 绑定后出现与柔体主题一致的绑定编辑区域。
- K/P 权重范围可独立编辑。
- 已绑定状态需要可见、可取消。

5. 运行时要求
- 渲染运行时必须真正应用这些 K/P 绑定后的数值变化。
- 不能只做 UI，不做实际效果。


八、原先未涉及绑定的参数也要补上 K/P 支持

1. 范围要求
- 不仅是原先已经绑定过的参数要拆分。
- 原先没有绑定能力、但明显适合绑定的参数，也要加上 K/P 支持。

2. 重点对象
- 经典填充参数
- 经典连线参数
- 柔体相关仍未覆盖的可调数值参数


九、预设文件兼容性要求

1. 修改参数结构后，旧预设不能直接失效
- 新的参数方案引入“基础值 + 绑定状态 + 权重范围”后，必须考虑旧预设兼容。
- 旧预设文件加载时不能因为缺新字段而报错。

2. 兼容目标
- 旧预设能正常加载。
- 没有的新字段应自动补默认值。
- 主题混合与分区预设结构也应保持兼容。

3. 建议保存结构
- 完整预设可包含：
  - 普通配置键
  - _preset_meta
  - _theme_mix
- 分区预设需要覆盖对应主题或分区所需的所有键，包括 K/P 绑定键。


十、分区 / 子预设要求

1. 分区预设命名与提示
- 各主题/子主题/分区的保存与读取提示，应该使用更自然的中文名称。
- 例如：
  - 保存经典主题
  - 保存填充子主题
  - 保存连线子主题
  - 保存柔体主题
  - 已读取颜色方案子预设

2. 分区预设覆盖键集
- 分区预设键集不能只保存旧字段。
- 必须覆盖该分区真实相关的全部字段。
- 对于 contour / bars / tentacle / 主题分区，还应包含对应的 K/P 绑定配置键。


十一、预设切换与颜色/主题的联动要求

1. 预设切换时
- UI 中当前预设名称、颜色块、主题启用状态，都应同步切换。

2. 主题状态同步
- 切换预设后，左侧主题树复选框要立刻同步到新预设状态。
- 不能出现配置已经变了，但左侧勾选状态没更新的情况。

3. 窗口层状态同步
- 重新加载配置后，窗口层按钮/状态也应同步。
- 不能出现配置是 top，但按钮显示不对的情况。


十二、今天已经明确过的实现优先级

如果需要按优先级重新做，建议顺序如下：

第一优先级
- 主题树启用逻辑
- “经典万花筒”改“经典”
- seg_spin 缺失修复
- 预设切换顺序化

第二优先级
- 顶部右上角三个按钮
- 颜色块同步更新
- 柔体参数平滑过渡

第三优先级
- 经典填充/连线重构为 K/P 绑定 UI
- 未绑定参数扩展 K/P 支持
- 预设和分区预设兼容补全


十三、重做时的注意事项

1. 不要只恢复 UI 文案
- 这次要求里很多是“功能逻辑”，不是纯界面改字。

2. 不要只做主界面，不改渲染侧
- 颜色同步、平滑过渡、K/P 绑定生效，这些都需要 UI 和渲染侧一起改。

3. 不要让旧预设报错
- 每次新增字段都要给默认值与兼容路径。

4. 不要重复把主题启用控件放回右侧面板
- 启用入口应固定放在左侧主题树。


十四、可作为验收标准的检查清单

- “经典万花筒”是否已全部改成“经典”相关表述。
- 左侧主题树是否存在右侧小复选框。
- 经典/填充/连线/柔体是否都可单独控制启用状态。
- 右侧页面是否不再重复放主题启用复选框。
- 预设管理是否有“上一个 / 下一个”。
- 自动切换是否按顺序轮播而不是随机乱跳。
- 无效预设文件是否会被跳过而不是导致整个列表异常。
- seg_spin 是否存在，程序启动后是否不再报该错误。
- 顶部右上角是否已有“置于顶层 / 置于底层 / 重启程序”。
- 切换预设时，颜色块是否立刻同步。
- 过渡过程中，颜色块是否持续变化而不是最后一下跳变。
- 柔体参数是否参与平滑过渡。
- 经典填充/连线参数是否具备柔体同款右键 K/P 绑定方式。
- K/P 绑定后的效果是否在运行时真实生效。
- 旧预设文件是否仍能加载且不报错。
